# CustomerOps Agent｜想法压力测试

## 1. 需求是否真实

**真实，但需要限定范围。**

售后客服工单处理是一个真实存在的业务场景。跨境电商和 3C 产品的客服团队每天处理大量重复性工单，涉及订单查询、物流追踪、退款判断、换货处理等环节。

**不是伪需求的原因**：
- 客服人员确实需要在多个系统之间切换（订单系统、物流系统、知识库）
- 新客服确实容易误判退款、换货规则
- 客服主管确实需要可追溯的处理记录
- 重复性工单确实占用了大量人力

**但要注意**：MVP 做的是"模拟"真实业务价值，不是"替代"真实客服系统。它的核心价值是展示技术能力，而不是真的要卖给电商公司。

**结论**：需求真实，但 MVP 的目标是技术展示而非商业落地。

## 2. 用户场景是否明确

**明确。**

| 维度 | 说明 |
|------|------|
| 用户是谁 | 跨境电商客服、3C 产品售后团队、客服主管 |
| 使用场景 | 客服收到一条售后工单，需要判断如何处理 |
| 用户输入 | 一条售后问题文本，可选订单号、产品类型 |
| 系统输出 | 结构化分析结果：意图、证据、工具查询结果、政策判断、回复建议、质检结果 |
| 谁来审核 | 客服主管或客服本人审核 AI 生成的建议 |

**场景示例**：
- 输入："我的订单 ORD-2024-001 已经 10 天没收到货了，要求退款"
- 系统自动完成：意图识别（物流延迟+退款）→ 检索物流政策 → 查询订单状态 → 判断是否符合退款条件 → 生成回复 → 质检

**结论**：场景清晰，输入输出明确。

## 3. 项目是否有差异化

### 和普通 Chatbot 的区别

| 维度 | 普通 Chatbot | CustomerOps Agent |
|------|-------------|-------------------|
| 架构 | 单模型对话 | 多 Agent 协作编排 |
| 流程 | 问答式 | 完整业务流程 |
| 工具调用 | 无 | 订单、物流工具 |
| 输出 | 自由文本 | 结构化输出 |
| 质检 | 无 | QA Agent |

### 和普通 RAG 文档问答的区别

| 维度 | 普通 RAG 问答 | CustomerOps Agent |
|------|-------------|-------------------|
| 目的 | 回答文档相关问题 | 处理业务工单 |
| 流程 | 检索 → 回答 | 分类 → 检索 → 查询 → 判断 → 回复 → 质检 |
| 工具 | 无 | 订单、物流工具 |
| 输出 | 文本回答 | 结构化处理建议 |

### 和普通客服 FAQ 的区别

| 维度 | 普通 FAQ | CustomerOps Agent |
|------|---------|-------------------|
| 匹配方式 | 关键词匹配 | 语义理解 + 业务流程 |
| 处理能力 | 只能回答预设问题 | 能处理复杂工单 |
| 工具调用 | 无 | 有 |
| 政策判断 | 无 | 有 |

### 和 ProfileAgent 的区别

| 维度 | ProfileAgent | CustomerOps Agent |
|------|-------------|-------------------|
| 场景 | 个人展示、求职问答 | 售后客服工单处理 |
| 核心能力 | 对话、展示 | 业务流程编排 |
| 复杂度 | 相对简单 | 多 Agent 协作 |

### 和 CodePilot 的区别

| 维度 | CodePilot | CustomerOps Agent |
|------|-----------|-------------------|
| 领域 | 代码开发辅助 | 售后客服工单处理 |
| 核心能力 | 代码分析、生成 | 业务流程编排 |
| Agent 类型 | 开发相关 Agent | 业务流程 Agent |
| RAG 内容 | 代码库 | 售后政策 |
| Tool Calling | 开发工具 | 业务工具 |
| 面试展示点 | AI 工程能力 | 业务流程理解 |

**关键结论**：CodePilot 是开发者工具 Agent，CustomerOps Agent 是业务流程 Agent。两者形成互补，不重复。

## 4. MVP 是否足够小

**需要检查。**

当前 MVP 范围：
- 6 个 Agent（Intent、Retrieval、Tool、Policy、Reply、QA）
- 1 个 Orchestrator
- 4 个 mock tools
- 5-6 篇知识库文档
- 1 个前端页面
- 基础 Eval

**评估**：2-4 天可以完成，但需要严格控制范围。

**如果过大，需要删减的**：
- 不做复杂的 Agent 重试逻辑
- 不做复杂的错误处理
- 不做复杂的前端交互
- 不做复杂的 Eval 指标
- 知识库文档内容保持精简

**结论**：MVP 范围可控，但必须严格收窄，不能膨胀。

## 5. 技术实现是否可控

| 模块 | 难度 | 原因 |
|------|------|------|
| Intent Agent | 简单 | 基于 prompt 的分类任务，输出结构化意图 |
| RAG Retriever | 中等 | 需要实现本地文档检索，可以用简单关键词匹配或轻量向量方案 |
| Tool Agent | 简单 | 调用 mock 函数，返回预设数据 |
| Policy Agent | 中等 | 需要根据政策 + 数据判断，逻辑相对复杂 |
| Reply Agent | 简单 | 基于上下文生成回复 |
| QA Agent | 简单 | 检查回复是否符合规范 |
| Orchestrator | 中等 | 编排 6 个 Agent 的执行顺序和数据传递 |
| Analyze API | 简单 | FastAPI 接口，调用 Orchestrator |
| Eval API | 简单 | 读取测试用例，执行评估 |
| 前端页面 | 中等 | 需要展示 Agent Timeline、Evidence、Tool Result |

**总体评估**：技术实现可控，没有明显的技术难点。核心挑战在于 Agent 编排和数据流转的设计。

## 6. 成本是否可接受

| 维度 | 评估 |
|------|------|
| LLM 调用成本 | MVP 阶段使用 mock 响应，成本为零。后续可选接入真实 LLM，单次工单处理成本约 0.01-0.05 美元 |
| 开发时间成本 | 2-4 天完成 MVP，可控 |
| 前端复杂度 | 中等，需要展示 Agent Timeline 和结构化结果 |
| 真实系统接入成本 | MVP 不接入真实系统，成本为零 |
| 部署成本 | MVP 本地运行，成本为零 |

**关键约束**：MVP 默认 mock-first，可选真实 LLM，不接真实业务系统。这使得成本完全可控。

## 7. 是否适合当前阶段开发

**适合。**

- 学生阶段：有足够时间完成 2-4 天的 MVP
- 实习求职：项目技术栈匹配 AI Agent 应用开发岗位
- 已有 CodePilot：CustomerOps Agent 与 CodePilot 形成差异化组合
- 简历包装：项目有明确的业务场景和技术深度

**风险**：如果不能在 2-4 天内完成，可能会影响求职时间线。需要严格控制范围。

## 8. 是否适合写进简历或作品集

**适合。建议作为第二核心项目。**

简历项目组合建议：

| 优先级 | 项目 | 定位 | 展示能力 |
|--------|------|------|---------|
| 第一 | CodePilot | 开发者工具 Agent | 代码分析、AI 工程、RAG、Tool Calling |
| 第二 | CustomerOps Agent | 业务流程 Agent | 多 Agent 编排、业务理解、结构化输出 |
| 辅助 | ProfileAgent | 个人展示 Agent | 前端展示、对话设计 |

**组合优势**：CodePilot 展示技术深度，CustomerOps Agent 展示业务理解，ProfileAgent 展示产品能力。三个项目覆盖 AI 应用开发的不同维度。

## 9. 是否存在明显伪需求

**严厉审查结果**：

| 问题 | 判断 | 说明 |
|------|------|------|
| 是否只是为了多 Agent 而多 Agent？ | 否 | 售后工单处理确实需要多个环节协作，多 Agent 是合理的技术选择 |
| 是否只是换皮客服机器人？ | 否 | 有完整的业务流程（分类→检索→查询→判断→回复→质检），不是简单问答 |
| 是否业务流程不够真实？ | 风险中等 | MVP 使用 mock 数据，但业务流程设计基于真实场景 |
| 是否没有必要做前端？ | 否 | 前端用于展示 Agent 执行过程，是面试演示的关键 |
| 是否没有必要做 Eval？ | 否 | Eval 是展示质量意识和迭代能力的重要环节 |

**需要修正的风险**：
- 业务流程需要足够真实，不能只是"调 API 返回固定数据"
- 知识库内容需要有实际价值，不能只是占位符
- Eval 需要有实际的 bad case，不能只是空测试

## 10. 是否存在过度设计风险

**重点检查以下功能是否应该进入 MVP**：

| 功能 | 是否进入 MVP | 原因 |
|------|-------------|------|
| 用户登录 | ❌ 不做 | MVP 不需要用户系统 |
| 多租户 | ❌ 不做 | MVP 只服务一个场景 |
| 权限系统 | ❌ 不做 | MVP 不需要权限控制 |
| 真实支付退款 | ❌ 不做 | MVP 使用 mock 数据 |
| 真实 Shopify / Amazon / 淘宝接口 | ❌ 不做 | MVP 不接真实业务系统 |
| 真实在线客服系统 | ❌ 不做 | MVP 只做分析建议 |
| 大规模向量数据库 | ❌ 不做 | MVP 使用本地 Markdown 文件 |
| 异步任务队列 | ❌ 不做 | MVP 使用同步处理 |
| 复杂长期记忆 | ❌ 不做 | MVP 不需要会话记忆 |
| 复杂知识库管理后台 | ❌ 不做 | MVP 直接读取本地文件 |
| 多语言客服 | ❌ 不做 | MVP 只做中文 |
| 自动发送真实客服消息 | ❌ 不做 | MVP 只生成建议，不发送 |

**必须删除的功能**：以上所有功能都不进入 MVP。

**只能作为后续扩展**：真实 LLM、真实数据库、真实电商 API、Docker 部署、多轮会话。

## 13. 压力测试结论

### 优势

1. **场景真实**：售后客服工单处理是真实业务需求
2. **技术匹配**：技术栈完全匹配 AI Agent 应用开发岗位
3. **差异化明显**：不是普通聊天机器人，是业务流程 Agent
4. **和 CodePilot 互补**：形成简历组合优势
5. **MVP 可控**：2-4 天可完成
6. **面试可讲解**：业务场景清晰，技术点丰富

### 风险

1. **业务流程可能不够真实**：使用 mock 数据，可能被质疑"只是 demo"
2. **知识库内容可能太简单**：如果只是占位符，没有实际价值
3. **Eval 可能流于形式**：如果没有真实的 bad case，评估没有意义
4. **前端可能过度设计**：如果花太多时间在前端，会挤占核心功能时间

### 必须收窄的地方

1. 不做任何真实业务系统接入
2. 不做复杂前端交互
3. 不做复杂 Eval 指标
4. 知识库内容保持精简但有实际价值
5. mock 数据要足够真实

### 不应该做的地方

1. 用户登录、权限、多租户
2. 真实支付、退款、电商 API
3. 复杂数据库、异步队列
4. 多语言、长期记忆
5. 生产级部署

### 推荐保留的 MVP 能力

1. 6 个 Agent + Orchestrator
2. 4 个 mock tools
3. 5-6 篇知识库文档
4. 基础前端展示（Agent Timeline + 结果）
5. 基础 Eval（10 条测试用例）
6. README 和文档

### 推荐删除的功能

1. 用户系统（登录、注册、权限）
2. 真实业务系统接入
3. 复杂前端交互
4. 复杂 Eval 指标
5. 异步任务队列
6. 长期记忆
7. 多语言支持
