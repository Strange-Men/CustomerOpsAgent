# CustomerOps Agent｜项目想法定义

## 1. 项目想解决什么问题

跨境电商和 3C 产品售后客服每天面对大量重复性工单：物流延迟、退款申请、换货请求、发票问题、产品故障。客服人员需要反复执行相同的动作——查订单状态、查物流轨迹、翻售后政策、判断是否符合条件、写回复话术。这个过程存在几个核心痛点：

**人工处理效率低**：一条工单平均需要 5-15 分钟，涉及多个系统切换（订单系统、物流系统、知识库、话术模板）。高峰期积压严重，客户等待时间长。

**新人误判风险高**：退款窗口期、换货条件、补偿规则散落在多份政策文档中。新客服容易遗漏关键条件，导致违规退款或错误拒绝，直接影响公司损失和客户满意度。

**AI 不能只聊天**：市面上的 AI 客服大多是问答机器人——用户问什么，AI 回什么。但售后工单不是闲聊，它需要按流程走：先识别意图，再查政策，再查订单，再判断能不能退，再生成回复。普通聊天机器人做不了这件事。

**结果需要可审核**：客服主管需要看到 AI 给出建议的依据——查了哪些政策、调用了什么数据、判断逻辑是什么。不能是一个黑盒输出。

**核心问题**：售后工单处理是一个结构化的业务流程，不是自由对话。现有方案缺少业务流程编排、工具调用、政策判断和质量检查的能力。

## 2. 目标用户是谁

| 用户角色 | 使用场景 | 核心需求 |
|---------|---------|---------|
| 跨境电商客服团队 | 日常处理售后工单 | 提高处理效率，减少重复劳动 |
| 3C 产品售后团队 | 处理电子产品退换货 | 准确判断政策，减少误判 |
| 中小电商运营人员 | 兼职客服处理售后 | 降低学习成本，快速上手 |
| 客服主管 | 监控客服质量 | 可追溯、可审核的处理记录 |
| 技术评估者 | 评估系统能力 | 多 Agent 编排、RAG、Tool Calling、工程化能力 |

**项目展示受众**：AI Agent 应用开发方向的技术评估者。关注的是"能不能设计和实现一个有业务逻辑的 Agent 系统"。

## 3. 用户为什么需要它

**效率**：一条工单从 10 分钟缩短到 2 分钟。系统自动完成意图识别、知识检索、订单查询、政策判断，客服只需要审核和发送。

**准确性**：系统严格按照政策规则判断，不会遗漏退款窗口期、不会搞混换货条件。每次判断都有政策依据。

**标准化**：所有客服的回复风格、话术模板、处理流程保持一致。不会出现"这个客服松、那个客服紧"的情况。

**可追溯**：每条工单的处理过程完整记录——查了什么政策、调了什么数据、判断逻辑是什么。主管可以随时审计。

**降低培训成本**：新客服不需要背诵所有政策，系统会自动提示处理建议和政策依据。从"人记规则"变成"系统执行规则"。

## 4. 它和普通方案有什么区别

### 普通 AI 客服 / FAQ 机器人

- 用户问什么，AI 回什么
- 没有业务流程，只是关键词匹配 + 模板回复
- 不调用订单、物流等外部工具
- 回复没有 evidence（政策依据）
- 没有质检环节
- 无法证明 Agent 工程能力
- 本质上是"套壳聊天机器人"

### CustomerOps Agent

- **先分类**：Intent Agent 识别工单意图（退款 / 换货 / 物流 / 发票）
- **再检索**：Retrieval Agent 从本地知识库检索相关政策文档
- **再查工具**：Tool Agent 调用 mock 订单 / 物流工具获取实时数据
- **再判断**：Policy Agent 根据政策 + 数据判断处理建议
- **再生成**：Reply Agent 生成专业客服回复
- **再质检**：QA Agent 检查回复风险和规范性
- **有 evidence**：每次判断都附带政策依据
- **有转人工**：低置信度或高风险工单自动建议转人工
- **有 Bad Case / Eval**：系统化的质量评估机制

**本质区别**：普通客服机器人是"问答系统"，CustomerOps Agent 是"业务流程自动化系统"。

## 5. 最小可行版本是什么

**MVP 只做一件事**：验证"一条售后工单能否通过多 Agent 流程完成分类、检索、工具查询、政策判断、回复生成和质检"。

具体包含：

1. **工单输入**：用户输入一条售后问题文本
2. **Intent Agent**：判断工单意图（退款 / 换货 / 物流 / 发票 / 投诉）
3. **Retrieval Agent**：从本地 Markdown 知识库检索相关政策
4. **Tool Agent**：调用 mock 订单 / 物流工具获取数据
5. **Policy Agent**：根据政策 + 数据判断处理建议
6. **Reply Agent**：生成客服回复
7. **QA Agent**：检查回复风险
8. **Orchestrator**：编排以上 Agent 的执行流程
9. **前端展示**：展示 Agent 执行过程和最终结果
10. **Eval / Bad Case**：基础评估能力

**MVP 不做**：真实 LLM、真实数据库、真实电商 API、用户登录、权限系统。

## 6. 这个项目最终希望证明什么能力

| 能力 | 体现方式 |
|------|---------|
| 多 Agent 编排能力 | 6 个 Agent 协作处理一条工单，有明确的流程和依赖 |
| RAG 检索能力 | 从本地知识库检索政策文档，提供 evidence |
| Tool Calling / Function Calling 能力 | 调用 mock 订单、物流工具，获取实时数据 |
| 结构化输出能力 | 每个 Agent 输出 Pydantic Schema 约束的结构化结果 |
| 后端 API 设计能力 | FastAPI 提供规范的 RESTful API |
| 前后端联调能力 | 前端展示 Agent 执行过程和结果 |
| Agent 评估迭代能力 | Bad Case 收集、Eval 指标、持续优化 |
| 业务场景理解能力 | 理解售后客服工单的完整业务流程 |

## 7. 为什么这个项目适合写进简历

**它不是套壳聊天机器人**。大多数 AI 项目只是"接个 API，套个前端"。CustomerOps Agent 有完整的业务流程设计、多 Agent 编排、RAG 检索、Tool Calling、结构化输出、QA 质检。这些是真实的 AI 工程能力。

**它有明确的业务场景**。不是"做一个聊天机器人"这种模糊需求，而是"售后客服工单处理"这个具体场景。面试时可以清晰地讲解业务背景、技术方案、实现细节。

**它有可量化的成果**。可以说"6 个 Agent 协作处理一条工单"、"5 篇知识库文档可检索"、"4 个 mock tools 可用"、"10 条 bad case 可评估"。

**它有技术深度**。涉及 Agent 编排、RAG、Tool Calling、结构化输出、FastAPI、Pydantic、前后端联调等技术栈。

## 8. 为什么适合深圳 AI Agent 应用开发类实习岗位

深圳 AI Agent 应用开发类岗位的常见关键词：

| 关键词 | CustomerOps Agent 如何体现 |
|--------|--------------------------|
| LangGraph / LangChain | 使用 LangGraph 编排多 Agent 流程 |
| Agent Workflow | 6 个 Agent 的完整工作流设计 |
| RAG | 从本地知识库检索政策文档 |
| Tool Calling / Function Calling | 调用订单、物流等 mock 工具 |
| FastAPI | Python 后端 API 框架 |
| Python 后端 | 完整的后端工程实现 |
| 数据库 / API 对接 | Mock 数据 + API 设计 |
| Prompt Engineering | 每个 Agent 的 prompt 设计 |
| Bad Case 分析 | 系统化的 Bad Case 收集和分析 |
| Benchmark / Eval | 基础评估指标和测试集 |
| 业务流程自动化 | 售后客服工单的完整业务流程 |
| 结构化输出 | Pydantic Schema 约束 LLM 输出 |

**匹配度**：CustomerOps Agent 覆盖了该类岗位 80% 以上的核心能力要求。它不是一个"玩具项目"，而是一个有业务深度的技术项目。
