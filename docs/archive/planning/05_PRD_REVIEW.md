# CustomerOps Agent｜PRD 审查报告

**审查版本**：v1.0
**审查日期**：2026-06-11
**审查对象**：docs/04_PRD_DRAFT.md（初版 PRD）
**审查角色**：产品经理、技术负责人、面试官

---

## 1. PRD 是否误解项目定位

### 审查结论：定位准确，未误解。

初版 PRD 明确了以下关键定位：
- ✅ 明确写了"CustomerOps Agent 不是什么"（不是聊天机器人、不是 FAQ、不是纯 RAG、不是完整 SaaS）
- ✅ 核心流程是固定的 6 Agent 线性编排，不是自由对话
- ✅ 输出是结构化的 JSON，不是自由文本
- ✅ 有 evidence 和政策依据，不是黑盒输出
- ✅ 有 QA Agent 质检，不是只生成不检查

### 潜在风险

**风险**：场景 5（投诉/高风险）中，回复生成可能偏向"安抚性话术"，容易让读者误以为是普通聊天机器人。

**建议**：在 PRD 中强调"回复内容受政策约束，不是自由生成"。

### 审查结论：PRD 未误解项目定位。

## 2. MVP 是否过大

### 审查结论：MVP 范围合理，但部分功能需保持轻量。

**必须保持轻量的功能**：

| 功能 | 当前描述 | 风险 | 建议 |
|------|---------|------|------|
| QA Agent | "检查回复质量、风险和规范性" | 可能被做成复杂的 NLP 质检系统 | MVP 只做规则检查：是否包含必要字段、是否超长、是否包含禁用词 |
| Policy Agent | "根据业务规则判断是否符合条件" | 可能被做成复杂的规则引擎 | MVP 只做简单条件判断：时间窗口 + 状态匹配 |
| Retrieval Agent | "从本地 Markdown 知识库检索" | 可能被做成向量检索 | MVP 只做关键词匹配 + 文件读取 |
| Orchestrator | "编排 6 个 Agent 的执行顺序" | 可能被做成动态路由 | MVP 固定线性流程，不做条件分支 |
| Eval | "10 条测试用例 + 评估指标" | 可能被做成完整的 eval 框架 | MVP 只做固定用例执行 + 简单对比 |
| 前端 | "Agent Timeline + 结果展示" | 可能被做成复杂的 SPA | MVP 用最简单的组件实现 |

### 审查结论：MVP 范围可控，但上述功能必须保持轻量实现，避免过度设计。

## 3. 哪些功能必须保留

### 支撑项目价值的核心功能

以下功能是项目的核心价值所在，**必须保留**：

| # | 功能 | 保留原因 |
|---|------|---------|

| 3 | **Retrieval Agent + 知识库** | 展示 RAG 能力，输出 evidence |
| 4 | **Tool Agent + mock tools** | 展示 Tool Calling 能力 |
| 5 | **Policy Agent** | 展示业务逻辑判断能力 |
| 6 | **Reply Agent** | 展示基于上下文的回复生成能力 |
| 7 | **QA Agent** | 展示质量保证意识，输出 need_human_review |
| 8 | **Orchestrator** | 展示多 Agent 编排能力 |
| 9 | **Analyze API** | 统一入口，展示 API 设计能力 |
| 10 | **前端 Agent Timeline** | 展示 Agent 执行过程，面试演示关键 |
| 11 | **结构化输出** | 全程 Pydantic Schema 约束 |
| 12 | **evidence 输出** | 每次判断附带政策依据 |
| 13 | **need_human_review** | 高风险工单自动标记转人工 |
| 14 | **10 条示例工单** | 演示和验证的基础 |
| 15 | **基础 Eval** | 展示质量意识 |

### 审查结论：以上 15 项功能必须保留，缺一不可。

## 6. Goal 模式开发风险

### 6.1 Claude 是否可能乱加登录权限

**风险等级**：高

**风险描述**：Goal 模式下，Claude 可能在实现过程中自动添加用户认证中间件、JWT token 验证、权限检查等。

**控制方式**：
- PROJECT_SCOPE.md 明确列出"禁止添加用户登录/权限系统"
- DEV_STATUS.md 更新禁止事项
- Goal 模式启动时必须读取 PROJECT_SCOPE.md

### 6.2 是否可能过度设计数据库

**风险等级**：中

**风险描述**：Claude 可能创建复杂的数据库模型、ORM 关系、迁移脚本。

**控制方式**：
- PROJECT_SCOPE.md 明确"使用 mock 数据和 JSON 文件，不引入数据库"
- MVP 不需要持久化存储
- 工单数据只在请求期间存在

### 6.3 是否可能把 Agent 写得太重

**风险等级**：中

**风险描述**：Claude 可能在单个 Agent 中加入过多逻辑（重试、缓存、降级、日志等）。

**控制方式**：
- 每个 Agent 保持单一职责
- Agent 内部逻辑不超过 100 行
- 不做重试、缓存、异步等复杂逻辑
- Orchestrator 只做线性编排

### 6.4 是否可能忽略测试

**风险等级**：中

**风险描述**：Goal 模式下 Claude 可能急于实现功能，跳过测试。

**控制方式**：
- 验收标准中包含"后端核心模块有测试"
- DEV_RULES.md 明确"测试覆盖率不低于 80%"
- 每个 Agent 和 API 接口都需要测试

### 6.5 是否可能做成聊天页面

**风险等级**：高

**风险描述**：前端实现时 Claude 可能做成对话式聊天界面，而不是 Agent Timeline + 结构化结果展示。

**控制方式**：
- PRD 明确前端必须有 Agent Timeline 组件
- PRD 明确结果展示页面的结构
- 前端设计文档（Module 4）将明确页面布局

### 6.6 是否可能破坏 API 契约

**风险等级**：中

**风险描述**：Claude 可能在实现过程中修改 API 输入输出格式，导致前后端不一致。

**控制方式**：
- PRD 明确定义输入输出 Schema
- 使用 Pydantic 模型约束 API 请求和响应
- API 变更必须更新文档

### 审查结论：Goal 模式风险需要通过 PROJECT_SCOPE.md 和 DEV_RULES.md 双重约束控制。

## 7. PRD 收窄建议

### 7.1 最终产品边界

**做**：
- 6 个 Agent 线性流程（Intent → Retrieval → Tool → Policy → Reply → QA）
- 1 个 Orchestrator（简单线性编排）
- 4 个 mock tools
- 6 篇 Markdown 知识库
- 1 个 Analyze API + 1 个 Eval API + 1 个 Health API
- 前端：工单输入 + Agent Timeline + 结果展示 + Eval 页面
- 10 条示例工单 + 10 条测试用例
- 基础测试

**不做**：
- 用户系统（登录、权限、多租户）
- 真实业务系统（支付、退款、电商 API）
- 复杂基础设施（数据库、向量库、消息队列）
- 复杂前端（移动端、复杂交互、聊天框）
- 生产级部署（Docker、CI/CD）

### 7.2 实现约束

| 维度 | 约束 |
|------|------|
| Agent | 每个 Agent 单一职责，内部逻辑不超过 100 行 |
| Orchestrator | 固定线性流程，不做条件分支和动态路由 |
| RAG | 关键词匹配 + 本地文件读取，不做向量检索 |
| Tool | mock 实现，返回预设数据，不执行真实操作 |
| Policy | 简单条件判断（时间窗口 + 状态匹配），不做规则引擎 |
| QA | 规则检查（必要字段、超长、禁用词），不做 NLP 质检 |
| 前端 | 最简单组件实现，不做复杂交互 |
| 测试 | 核心模块单元测试，不做复杂集成测试 |
| Eval | 固定用例执行 + 简单对比，不做自动评估框架 |

### 7.3 收窄后的产品定位

CustomerOps Agent 是一个**轻量级、展示型**的多 Agent 业务流程 MVP，核心价值是：

1. 展示多 Agent 编排能力
2. 展示 RAG 检索能力
3. 展示 Tool Calling 能力
4. 展示结构化输出能力
5. 展示业务流程理解能力

**不是**一个完整的客服系统，**不是**一个可部署的 SaaS 产品。
