# M8 Answer Quality Evaluation Report

## 1. Evaluation Scope

M6 evaluated **retrieval quality** — whether the retriever finds the right documents.

M8 evaluates **answer quality** — whether the agent workflow produces good answers based on retrieved evidence.

Important context:
- Current answers are generated by a **mock answer generator** (rule-based templates), not a real LLM.
- Current logistics tool is a **mock logistics tool**, not a real API.
- M8 establishes the answer quality **baseline** for the current workflow.

## 2. Metrics

| Metric | Definition |
|--------|-----------|
| **Relevance** | Whether the response addresses the user's question (category match, route correctness, non-empty answer) |
| **Groundedness** | Whether the response is grounded in evidence (citations for RAG route, tool_used for logistics route) |
| **Completeness** | Whether the response covers expected keywords from the eval case |
| **Citation Hit Rate** | Fraction of cases where citation doc_ids match expected doc_ids |
| **Answer Pass Rate** | Fraction of cases passing all thresholds (relevance ≥ 0.6, groundedness ≥ 0.7, completeness ≥ 0.4, RAG citation hit = 1.0, logistics tool_used required) |
| **Fallback Rate** | Fraction of cases where fallback was triggered |

## 3. Dataset

- **Source**: `backend/data/eval_cases_full.jsonl`
- **Total cases**: 122
- **Category coverage**: logistics(18), customs(16), package(18), return(10), refund(10), exchange(10), address(10), order(10), payment(10), coupon(10)
- **Market coverage**: US(37), EU(16), GLOBAL(69)
- **Language coverage**: zh(82), en(40)
- **Difficulty coverage**: easy(16), medium(75), hard(31)
- `expected_doc_ids` used for citation hit evaluation
- `expected_keywords` used for completeness evaluation

### Failure Pattern Summary

1. **unknown_intent fallback (40.16%)**: Many cases trigger fallback because the intent recognizer routes them to "other" instead of the correct intent. This is the dominant failure mode.

2. **missing_order_id fallback**: Logistics-related questions without an explicit order ID fall back, even when the question is about policy (not tracking).

3. **Low completeness on RAG cases**: Even when RAG route is correct, the mock answer generator's template may not include expected keywords from the eval case.

## 6. Interpretation

### Key Findings

1. **Answer pass rate (31.97%) is much lower than retrieval recall (98.36%)**. This confirms that good retrieval does not guarantee good answers. The gap between M6 retrieval metrics and M8 answer metrics is significant.

2. **Fallback rate (40.16%) is the primary bottleneck**. Nearly half the cases fall back, mostly due to intent recognition routing issues. The intent recognizer classifies many aftersale/logistics questions as "other", triggering the fallback path.

3. **Completeness (0.34) is the weakest metric**. The mock answer generator uses fixed templates that don't cover the diverse expected_keywords in the eval set. This is expected for a rule-based system.

4. **Groundedness (0.70) is moderate**. When RAG route succeeds with citations, groundedness is high. But fallback cases pull the average down.

5. **Citation hit rate (56.56%)** reflects that only ~60% of cases go through the RAG route with valid citations. The rest are fallback or logistics tool cases.

### What M8 Does NOT Mean

- These results do **not** indicate the knowledge base is inadequate. M6 proved retrieval is 98.36% Recall@5.
- These results do **not** indicate the RAG design is flawed. The issue is in routing (intent recognition) and answer generation (mock templates).
- The mock answer generator is intentionally simple. A real LLM would produce much better answers.

### Next Steps (M9)

M9 should focus on:
1. Fixing intent recognition to reduce unknown_intent fallback
2. Improving mock answer templates to better cover expected keywords
3. Adjusting fallback rules for policy questions that don't need order IDs
4. Evaluating whether RAG citation coverage can be improved
