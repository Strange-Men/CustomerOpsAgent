# CustomerOps Agent｜开发规则

## 1. 总开发原则

### 1.1 小步快跑原则
- **每轮只做一个小任务**，不要一次性开发完整项目
- 每轮修改前先明确：目标、修改范围、禁止范围、验收标准
- 每轮结束后输出：完成内容、未完成内容、当前状态、下一步建议

### 1.2 渐进式开发原则
- 从最简单的可运行版本开始
- 逐步增加功能，每步都可验证
- 优先保证 mock 模式可运行

### 1.3 文档先行原则
- 重要决策必须记录在文档中
- API 变更必须更新文档
- 代码变更必须更新 CHANGELOG

### 1.4 Goal 模式开发总规则

后续 Goal 模式开发**必须遵守**以下规则：

| # | 规则 | 说明 |
|---|------|------|
| 1 | 只实现 PRD.md 的 Current MVP Scope | 不得自行扩大范围 |
| 2 | 不允许提前实现 Future Possible Scope | 这些是后续版本的事 |
| 3 | 不允许突破 Permanent Safety Boundaries | 任何版本都不能突破 |
| 4 | 不允许把项目做成普通聊天机器人页面 | 重点是 Agent 工作流和结构化结果 |
| 5 | 不允许把核心逻辑堆到一个大文件 | 单个文件不超过 300 行 |
| 6 | 不允许一次性无边界重构 | 小步快跑，每次改动可控 |
| 7 | 不允许改无关文件 | 只修改本轮目标范围内的文件 |
| 8 | 不允许删除已有测试 | 只能新增或修改测试 |
| 9 | 不允许破坏 API / Schema 契约 | 接口和数据结构变更必须先更新文档 |
| 10 | 每次开发必须保持可运行、可测试、可回滚 | 不允许留下不可运行的中间状态 |
| 11 | 如需扩展，必须先更新 PRD.md | 不得自行扩大范围 |

## 2. AI 协作规则

### 2.1 上下文管理
- 每轮开始前提供清晰的任务描述
- 明确当前项目状态和限制
- 提供足够的背景信息

### 2.2 任务拆解
- 复杂任务必须拆解为多个小任务
- 每个小任务独立可验证
- 明确任务之间的依赖关系

### 2.3 反馈机制
- 每轮结束后提供明确的验收结果
- 指出问题和改进建议
- 记录决策原因

### 2.4 Claude / AI 开发前必读文档

后续 Claude 开发前**必须先读取**以下文档：

| 顺序 | 文档 | 说明 |
|------|------|------|
| 1 | `docs/PROJECT_CONTEXT.md` | 项目背景和定位 |
| 2 | `docs/PRD.md` | 产品需求文档（MVP 范围、安全边界、非目标） |
| 3 | `docs/DESIGN.md` | 设计文档（页面、组件、用户流程） |
| 4 | `docs/TECHNICAL_SPEC.md` | 技术规格文档（技术栈、API 契约、Schema、目录结构） |
| 5 | `docs/DEV_RULES.md` | 开发规则（本文档） |
| 6 | `docs/DEV_STATUS.md` | 开发状态（当前阶段、已完成、下一步） |

`docs/archive/` 目录只作为**历史参考**，不作为默认上下文。除非需要回溯决策过程，否则不需要读取。

### 2.5 Claude 每轮输出要求

Claude 每轮开发**必须输出**以下内容：

| # | 输出项 | 说明 |
|---|--------|------|
| 1 | 本轮目标 | 本轮要完成什么 |
| 2 | 修改文件 | 本轮修改了哪些文件 |
| 3 | 未做事项 | 本轮没有做什么，为什么 |
| 4 | 测试结果 | 测试是否通过，覆盖率 |
| 5 | Git 状态 | working tree 是否 clean |
| 6 | 下一步建议 | 下一步应该做什么 |

## 3. 每轮开发规则

### 3.1 开始前
- [ ] 明确本轮目标
- [ ] 明确修改范围（允许修改的文件）
- [ ] 明确禁止范围（不允许修改的文件）
- [ ] 明确验收标准
- [ ] 确认当前项目状态

### 3.2 开发中
- [ ] 只修改允许范围内的文件
- [ ] 每个文件修改前先 Read 确认当前内容
- [ ] 保持代码风格一致性
- [ ] 及时记录重要决策

### 3.3 结束后
- [ ] 验收所有标准
- [ ] 输出本轮总结
- [ ] 更新 DEV_STATUS.md
- [ ] 更新 CHANGELOG.md
- [ ] 提供下一步建议

## 4. 文件修改规则

### 4.1 修改前
- 必须先 Read 文件，确认当前内容
- 确认文件在允许修改范围内
- 确认修改不会破坏现有功能

### 4.2 修改中
- 使用最小修改原则，不要大范围重构
- 保持现有代码风格
- 添加必要的注释
- 确保修改是原子性的

### 4.3 修改后
- 验证修改是否正确
- 确认没有引入新问题
- 更新相关文档

### 4.4 分层职责规则

每层只做自己的事，不允许跨层写业务逻辑：

| 层 | 职责 | 禁止 |
|----|------|------|
| API 层 | 只处理请求和响应 | 不写业务逻辑 |
| Service 层 | 负责编排业务流程 | 不写 Agent 内部逻辑 |
| Agent 层 | 负责 Intent / Retrieval / Tool / Policy / Reply / QA | 不写 API 处理逻辑 |
| RAG 层 | 只负责知识库加载和检索 | 不写业务判断 |
| Tools 层 | 只负责 mock 工具 | 不写业务逻辑 |
| Schemas 层 | 只负责数据结构 | 不写业务逻辑 |
| Core 层 | 只负责配置、错误、日志、LLM client | 不写业务逻辑 |
| Frontend components | 只负责 UI 展示 | 不写后端业务逻辑 |

## 5. 模块拆分规则

### 5.1 分层架构
```
API 层 (api/)
├── 只负责请求和响应
├── 不写复杂业务逻辑
└── 调用 Service 层

Service 层 (services/)
├── 负责业务编排
├── 协调多个 Agent 和 Tool
└── 处理业务流程

Agent 层 (agents/)
├── 负责智能决策
├── 每个 Agent 有明确职责
└── 输出结构化结果

RAG 层 (rag/)
├── 负责知识检索
├── 管理知识库
└── 提供检索结果

Tools 层 (tools/)
├── 负责工具调用
├── 封装外部 API
└── 提供统一接口

Schemas 层 (schemas/)
├── 负责数据结构
├── Pydantic 模型定义
└── 数据验证
```

### 5.2 文件组织
- 每个模块一个目录
- 每个文件职责单一
- 避免大文件（单个文件不超过 300 行）
- 相关功能放在同一目录

### 5.3 命名规范
- 文件名：小写字母 + 下划线（snake_case）
- 类名：大驼峰（PascalCase）
- 函数名：小写字母 + 下划线
- 常量：大写字母 + 下划线

## 6. API 契约规则

### 6.1 请求规范
- 使用 Pydantic 模型定义请求体
- 明确必填和可选字段
- 提供字段验证和错误提示

### 6.2 响应规范
- 统一响应格式：`{code, message, data}`
- 成功响应：`code=200`
- 错误响应：明确错误码和错误信息
- 使用 Pydantic 模型定义响应体

### 6.3 错误处理
- 捕获所有可能的异常
- 提供友好的错误信息
- 记录错误日志
- 不暴露内部实现细节

### 6.4 文档要求
- 使用 FastAPI 自动生成的 OpenAPI 文档
- 提供请求和响应示例
- 说明业务逻辑和限制

## 7. 安全规则

### 7.1 API Key 管理
- API Key 只能放后端环境变量
- 前端不能出现任何密钥
- .env 不能提交 Git
- 使用 .env.example 说明需要的环境变量

### 7.2 输入验证
- 所有用户输入必须验证
- 防止 SQL 注入、XSS 等攻击
- 限制输入长度和格式

### 7.3 输出安全
- 不暴露内部错误堆栈
- 不泄露敏感信息
- 控制输出数据范围

## 8. LLM 使用规则

### 8.1 结构化输出
- LLM 输出必须经过结构化校验
- 使用 Pydantic 模型定义输出格式
- 处理 LLM 输出不符合预期的情况

### 8.2 业务安全
- LLM 输出不能直接执行真实业务动作
- 必须经过业务逻辑验证
- 重要操作需要人工确认

### 8.3 Mock 优先
- mock 模式必须优先可运行
- 开发阶段使用 mock 响应
- 逐步替换为真实 LLM

### 8.4 错误处理
- 处理 LLM 调用失败的情况
- 设置合理的超时时间
- 提供降级方案

## 9. 测试规则

### 9.1 单元测试
- 核心业务逻辑必须有单元测试
- 测试覆盖率不低于 80%
- 测试用例要覆盖正常和异常情况

### 9.2 集成测试
- API 接口必须有集成测试
- 测试完整的业务流程
- 测试 Agent 之间的协作

### 9.3 Bad Case 测试
- 收集和记录 Bad Case
- 建立 Bad Case 测试集
- 定期回归测试

### 9.4 测试原则
- 测试先行（TDD）
- 测试独立，不依赖外部服务
- 测试可重复执行

## 10. Git 策略规则

### 10.1 提交规范
- 使用 Conventional Commits 规范
- 每个提交只做一件事
- 提交信息清晰描述变更

### 10.2 Git Checkpoint 规则

项目开发过程中**必须在以下节点执行 Git commit**：

| # | 节点 | 说明 |
|---|------|------|
| 1 | 文档阶段完成后 | 每个 Module 文档完成后必须 commit |
| 2 | 项目初始化前 | 必须保证 working tree clean |
| 3 | 项目初始化完成后 | 初始化结构完成后必须 commit |
| 4 | Goal 模式 MVP 开发前 | 必须 commit 当前初始化状态 |
| 5 | Goal 模式 MVP 完成后 | MVP 代码完成后必须 commit |
| 6 | MVP 验收通过后 | 打 tag：`v0.1-mvp` |
| 7 | 大改动前 | 新建分支 |
| 8 | 修 bug 前 | 先确认当前状态可回滚 |

**禁止**：
- 不允许在未提交状态下连续进行多轮大改
- 不允许在 working tree dirty 时开始新的大功能

### 10.3 推荐提交信息

| 场景 | 提交信息 |
|------|---------|
| 文档更新 | `docs: update development rules and init checklist` |
| 项目初始化 | `chore: initialize project structure` |
| MVP 完成 | `feat: complete customerops agent mvp` |
| Bug 修复 | `fix: <描述>` |
| 测试相关 | `test: <描述>` |
| 文档相关 | `docs: <描述>` |
| 配置相关 | `chore: <描述>` |
| 重构 | `refactor: <描述>` |

### 10.4 分支管理
- main 分支保持稳定
- 开发在 feature 分支进行
- 合并前必须通过测试
- 大改动前新建分支

### 10.5 回滚策略
- 每个功能点独立提交
- 出现问题可以快速回滚
- 保留回滚说明
- 修 bug 前先确认当前状态可回滚

## 11. 禁止事项

### 11.1 开发禁止
- ❌ 不允许一次性开发完整项目
- ❌ 不允许把核心逻辑写进一个大文件
- ❌ 不允许盲目大重构
- ❌ 不允许跳过测试
- ❌ 不允许无 evidence 承诺退款或换货

### 11.2 安全禁止
- ❌ 前端不能出现任何密钥
- ❌ .env 不能提交 Git
- ❌ LLM 输出不能直接执行真实业务动作
- ❌ 不允许暴露内部错误堆栈

### 11.3 范围禁止
- ❌ 不接真实电商平台
- ❌ 不做真实退款、真实支付
- ❌ 不做用户登录、权限系统
- ❌ 不做多租户架构
- ❌ 不做复杂数据库
- ❌ 不做实时消息推送
- ❌ 不做复杂的前端交互（MVP 阶段）
- ❌ 不接入真实 LLM（初期使用 mock 响应）
- ❌ 不做生产级部署

### 11.4 开发环境禁止
- ❌ 不允许在 base 环境直接安装项目依赖
- ❌ 不允许使用系统全局 Python 安装项目依赖
- ❌ 不允许默认使用 venv，必须使用 Conda
- ❌ 不允许手动安装依赖但不写入 requirements.txt 或 environment.yml

## 12. Python 开发环境规则（Conda）

### 12.1 Conda 环境要求
- 后端 Python 开发**必须先激活 Conda 环境**
- 推荐环境名：`customerops-agent`
- 推荐 Python 版本：3.11
- 禁止在 base 环境或系统全局 Python 中安装项目依赖

### 12.2 依赖管理
- 所有依赖必须记录在 `requirements.txt` 或后续 `environment.yml` 中
- 不能只手动 pip install 不记录
- 安装依赖前必须确认已激活正确的 Conda 环境

### 12.3 Claude 协作规则
- Claude 运行 Python 命令前，必须先确认当前 Conda 环境
- 若环境出现问题，优先检查：Conda 环境 → Python 版本 → 依赖版本
- 不要盲目改代码，先排查环境问题

### 12.4 常用命令参考

```bash
# 创建环境
conda create -n customerops-agent python=3.11

# 激活环境
conda activate customerops-agent

# 安装依赖（在 backend/ 目录下）
pip install -r requirements.txt

# 运行测试
pytest

# 代码检查
ruff check .

# 启动后端
uvicorn app.main:app --reload
```

## 13. 禁止提交内容

以下内容**绝对不允许提交到 Git**：

| # | 禁止提交 | 说明 |
|---|---------|------|
| 1 | `.env` | 环境变量文件，包含密钥 |
| 2 | API Key | 任何密钥、Token |
| 3 | 密钥 | 密码、私钥、证书 |
| 4 | `node_modules/` | Node.js 依赖 |
| 5 | `venv/` | Python 虚拟环境 |
| 6 | `.next/` | Next.js 构建缓存 |
| 7 | `__pycache__/` | Python 字节码缓存 |
| 8 | `.pytest_cache/` | pytest 缓存 |
| 9 | `dist/` | 构建输出 |
| 10 | `build/` | 构建输出 |
| 11 | 临时文件 | `*.tmp`, `*.bak`, `*.swp` |
| 12 | 本地 IDE 缓存 | `.idea/`, `.vscode/settings.json` |

以上内容必须在 `.gitignore` 中配置。
